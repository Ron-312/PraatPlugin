# ADR-004: Store Temp Files Under TMPDIR/PraatPlugin/<UUID>/

**Date:** 2026-03-25
**Status:** Accepted
**Deciders:** Project founder

---

## Context

PraatRunner requires its input audio on disk as a WAV file before it can call Praat. The plugin must:

1. Write a WAV file to a known location
2. Pass that path to Praat via CLI argument
3. Optionally receive output files back from Praat (if the script writes files rather than printing to stdout)
4. Clean up those files after the job completes

Questions to answer:
- Where should temp files go?
- How should files be named to avoid collisions between concurrent plugin instances or rapid sequential jobs?
- When and by whom should files be deleted?

## Decision

All temp files are stored under:
```
$TMPDIR/PraatPlugin/<job-uuid>/
    input.wav
```

- `$TMPDIR` is resolved using `juce::File::getSpecialLocation(juce::File::tempDirectory)`, which maps to `/var/folders/<hash>/T/` on macOS (per-user, per-session temp directory managed by the OS).
- `<job-uuid>` is a UUID generated by `juce::Uuid` at job creation time.
- Each job gets its own directory to avoid filename collisions between concurrent jobs or rapid sequential runs.
- The directory and all its contents are deleted by `JobDispatcher` immediately after the job completes or fails.

## Rationale

**Why TMPDIR:**
- `juce::File::tempDirectory` resolves correctly on all JUCE-supported platforms.
- On macOS, `/var/folders/.../T/` is per-user (no permission issues), writable without elevated privileges, and automatically cleaned by the OS on reboot.
- Standard convention: temporary data that doesn't need to survive a restart belongs in TMPDIR.

**Why a UUID per job:**
- Multiple instances of PraatPlugin can run simultaneously in a DAW (one per track).
- The user may click Analyze multiple times in rapid succession before a previous result is cleaned up.
- A UUID per job directory guarantees no two jobs compete for the same filename.

**Why delete immediately after the job:**
- Temp files can be large (30 seconds of stereo 48 kHz audio ≈ 11 MB per WAV).
- Leaving them accumulates disk usage silently.
- macOS cleans TMPDIR on reboot, but that's an unreliable backup; we should clean up ourselves.

**Why not use App Support or Documents:**
- These locations are for persistent user data. Temp analysis WAV files are not user data.
- Using App Support for scratch files would require implementing our own periodic cleanup.

## Alternatives Considered

| Option | Why not chosen |
|--------|---------------|
| `/tmp/PraatPlugin/` | `/tmp` on macOS is a symlink to `/private/tmp` and works, but `juce::File::tempDirectory` is more portable and idiomatic. |
| Fixed filename (`/tmp/PraatPlugin_input.wav`) | Collides between concurrent plugin instances or if previous run didn't clean up. |
| NSFileManager temporary directory API | Objective-C/macOS only; `juce::File::tempDirectory` is cross-platform and equivalent. |
| Keeping temp files for inspection | Useful for debugging but creates unbounded disk usage in production. Add a debug flag if needed. |

## Consequences

**Positive:**
- No disk accumulation: each job is self-contained and self-cleaning
- No permission issues on macOS
- Concurrent instances don't interfere

**Negative / Trade-offs:**
- If the DAW crashes with a job in flight, temp files are orphaned in TMPDIR until the next OS reboot. This is acceptable — macOS will eventually clean them.
- If Praat writes output files to the job directory (some scripts may), those are also cleaned up. If the user wants to keep Praat output files, the script should write them to a user-specified output path, not the temp directory.

**Risks and mitigations:**
- Risk: `JobDispatcher` crashes before cleanup, leaving temp files.
  Mitigation: macOS reclaims TMPDIR on session end. For extra safety, `PraatPluginProcessor`'s destructor scans and deletes any `PraatPlugin/` subdirectories older than 1 hour.
- Risk: Disk is full, WAV write fails.
  Mitigation: `WavFileWriter` checks for write errors and returns a `WriteResult::FileWriteFailed` status, which propagates to an error message in the plugin UI.
